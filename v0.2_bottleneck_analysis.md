# v0.2 Bottleneck Analysis

> **Status:** Final
> **Method:** Runtime JIT counters (`jit_entry_count`, `jit_bailout_count`) measured in release mode for each headline benchmark. Cross-referenced with `sample` profiling and Criterion wall-clock times.
> **Date:** 2026-06-26

---

## 1. Summary

Each headline benchmark has a different bottleneck. There is no single "fix everything" item. The v0.2 priorities must be benchmark-specific:

| Benchmark | Time | JIT entries | Bailouts | Bottleneck | Lever |
|---|---|---|---|---|---|
| `jit_hot_function_1M` | 116 ms | 999,952 | 0 | Function call (BLR) overhead | Phase F inlining |
| `loop_sum_smi_1M` | 110 ms | 1 | 0 | Trace body execution (11 opcodes/iteration) | No clear lever |
| `proto_chain_lookup_5deep_1M` | 116 ms | 1 | 0 | Proto chain walk (5 extra loads per iteration) | Full-trace JIT coverage |
| `poly_prop_10shapes_1M` | 804 ms | **0** | 0 | **Interpreter dispatch (JIT never runs)** | Vector IC (Option B) |
| `array_push_grow_100k` | 53 ms | **0** | 0 | Interpreter dispatch (ArrayPush not in JIT) | ArrayPush JIT coverage |

## 2. Detailed Findings

### 2.1 jit_hot_function_1M — 116ms, gap 37×

**JIT stats:** 999,952 entries, 0 bailouts. Every call to `add(a,b)` enters the function JIT from within the trace.

**Bottleneck:** The `BLR` (Branch with Link to Register) round-trip between trace code and function JIT code. Each iteration does:
1. Trace executes: load `s`, load `i`, set up args on JIT stack
2. Call helper: copy args, set up locals buffer, BLR to function JIT
3. Function JIT: untag Smis, add, check overflow, retag, push result
4. Return to trace: pop result, continue loop

The BLR + call helper overhead dominates. **This is the Phase F target.**

**Vec-fix impact:** 120ms → 116ms (within noise). The per-call Vec allocation was not the bottleneck — the function call entry/exit is.

### 2.2 loop_sum_smi_1M — 110ms, gap 43×

**JIT stats:** 1 entry, 0 bailouts. Single trace covers all 1M iterations.

**Bottleneck:** The trace body itself. Each iteration dispatches ~11 JIT-compiled opcodes:
- `LoadLocal s`, `LoadLocal i`, `Add`, `StoreLocal s`
- `LoadLocal i`, `LoadSmi 1`, `Add`, `StoreLocal i`
- `LoadLocal i`, `LoadSmi 1000000`, `Lt`, `JumpIfTrue`

At ~10ns per JIT opcode, that's ~110ns per iteration. The gap to V8 (2.3ms, 47× faster) suggests V8 eliminates loop overhead entirely (loop rotation, induction variable elimination).

**Lever:** No clear single fix. The gap is spread across 11 opcodes. Full optimization would require SSA-based optimization (the deferred `rune_jit_cranelift` tier). For v0.2, this benchmark is not the priority.

### 2.3 proto_chain_lookup_5deep_1M — 116ms, gap 68×

**JIT stats:** 1 entry, 0 bailouts. Single trace covers all 1M iterations.

**Bottleneck:** The proto chain walk adds 5 `ldr` instructions per iteration (walking 5 prototypes via `JSObject.prototype` at offset 24). Plus the property load at the end. This is ~116ns per iteration vs 106ms before the fixes (when the IC never worked). The remaining gap to V8 (1.55ms, 68× faster) is from:
- Proto chain walk in-JIT (5 loads + property load): ~15ns
- Loop overhead (loop counter, comparison, branch): ~10ns
- **Interpreter dispatch for other loop opcodes:** ~90ns

The proto chain walk itself is fast (ldr from known offset × 5 = ~2.5ns). The remaining time is the non-property-access parts of the loop body. Full-trace JIT covering all loop opcodes (already done — the trace covers them all) is the path forward. This benchmark is not actionable for v0.2 unless we can eliminate the non-property-access overhead.

### 2.4 poly_prop_10shapes_1M — 804ms, gap 173× ⚠️

**JIT stats:** **0 entries, 0 bailouts.** The JIT never successfully executes a trace for this benchmark.

**Root cause:** The trace recorder observes one shape during recording, burns it into the trace, and on the very next iteration the next object has a different shape → bail. The trace → bail → re-record → bail cycle oscillates without ever stabilizing. The recorder doesn't get enough consecutive same-shape iterations to reach steady-state.

**Impact:** 100% of iterations run in the interpreter. The 804ms is pure interpreter dispatch at ~800ns per iteration.

**Lever:** This is the highest-leverage benchmark. Getting the JIT to cover even 80% of iterations (via vector IC with N=8) would move from "0% JIT coverage" to "~80% JIT coverage", dropping runtime from 804ms to ~280-400ms. The vector IC (Option B from `multi_shape_dispatch_design.md`) is the right fix.

### 2.5 array_push_grow_100k — 53ms, gap 6.5×

**JIT stats:** 0 entries, 0 bailouts. `ArrayPush` is a bail-to-interpreter opcode (not in the JIT whitelist).

**Bottleneck:** Pure interpreter dispatch. The 53ms is all interpreter.

**Lever:** ArrayPush JIT coverage would move from 0% to ~100% JIT coverage, potentially dropping to ~25ms. But this is the smallest gap (6.5×) — lowest ROI for a P0 week.

## 3. Non-Finding: GC is Not a Bottleneck

None of the profiles showed significant time in GC (`SemiSpace::collect` or related symbols). The `--gc-stats` output (if added) would confirm, but the `Instruments` sample profile showed zero samples in GC code. For these benchmarks, Cheney semispace GC is fast enough — the allocation rate is modest (a few objects per outer eval, not per iteration).

## 4. Updated v0.2 Priority Recommendations

| Rank | Item | Target benchmark | Expected impact | Effort |
|---|---|---|---|---|
| **P0** | Phase F inlining | `jit_hot_function_1M` | 116ms → ~30-50ms | 2-3 weeks |
| **P0** | Vector IC (N=8) | `poly_prop_10shapes_1M` | 804ms → ~280-400ms | 1 week |
| **P1** | Full JIT coverage of remaining opcodes | All | Marginal per-opcode | 2+ weeks |
| **P1** | ArrayPush JIT | `array_push_grow_100k` | 53ms → ~25ms | 1 week |
| **P2** | float64 Sub/Mul/Div | (unblocks workloads) | Low | 1 week |
| **P2** | GenImmix GC spike | (GC-heavy benches) | Low (GC not the bottleneck) | 2 weeks |

**Critical note:** Phase F and vector IC are **independent** — they target different bottlenecks in different benchmarks. They can be worked in parallel. Phase F has the bigger absolute impact (116ms → ~30ms, 86ms saved) but is more effort. Vector IC has smaller absolute impact (804ms → ~400ms, 400ms saved) but is less effort and higher urgency (the JIT currently CANNOT run poly_prop at all).

**Recommendation:** Start vector IC first (1 week) — it changes poly_prop from "JIT non-functional" to "JIT functional for ≤8 shapes." Then Phase F (2-3 weeks) for the function-call overhead.

## 5. Methodology

JIT counters were collected via a standalone `rune_embed` example that calls `ctx.vm().jit_entry_count` and `ctx.vm().jit_bailout_count` after executing each benchmark's JS source once (with 1M iterations for the non-poly benchmarks, 100K for poly_prop to fit in a single eval).

The `sample` profiler was attached to a Criterion bench run of `jit_hot_function_1M` for ~10 seconds of wall-clock sampling. Results confirmed that the non-Criterion time inside `Vm::execute` is dominated by the interpreter warmup phase (~50 iterations) and the JIT function entry/exit overhead.

Counters were verified against the benchmark structure:
- `jit_hot_function_1M`: 1M calls to `add()` - 48 warmup calls = 999,952 JIT entries ✓
- `loop_sum_smi_1M`: 1 trace entry for the entire 1M iteration loop ✓
- `proto_chain_lookup_5deep_1M`: 1 trace entry, 0 bailouts ✓
- `poly_prop_10shapes_100K`: 0 entries — every trace recording bails on first non-recording iteration ✓
