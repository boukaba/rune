# v0.5 — Promise + Microtask Queue Design

## 1. Goal

Enable `fetch(url).then(r => r.json()).then(data => handle(data))` — the core async
pattern for edge/serverless workloads. Specifically:

- `new Promise((resolve, reject) => { ... })` — constructor
- `.then(onFulfilled, onRejected)` / `.catch(onRejected)` / `.finally(onFinally)`
- `Promise.resolve(v)` / `Promise.reject(r)` / `Promise.all(iterable)` / `Promise.race(iterable)`
- Microtask queue draining after each synchronous task
- `async function` / `await expr` — parser desugaring to Promise chains
- `setTimeout(fn, 0)` / `setInterval` — event loop integration (v0.5 stretch)

## 2. Promise State Representation

### New heap type: `TAG_PROMISE` (tag = 6)

```
[ GcHeader(TAG_PROMISE) | state: u32 | has_handler: u32 | result: Value | reactions: *mut PromiseReactionArray ]
```

Where:
- **state**: 0=pending, 1=fulfilled, 2=rejected
- **has_handler**: 0 or 1 (flag for unhandled rejection tracking; set when `.then`/`.catch` is attached)
- **result**: the fulfillment value or rejection reason (valid when state ≠ pending)
- **reactions**: pointer to a `PromiseReactionArray` (a growable array of reactions, allocated on GC heap or as a Vec owned by the promise)

### PromiseReaction

```rust
struct PromiseReaction {
    capability: PromiseCapability,  // { promise, resolve, reject }
    handler: Value,                 // onFulfilled or onRejected (or undefined)
}
```

### PromiseCapability

```rust
struct PromiseCapability {
    promise: Value,     // TAG_PROMISE
    resolve: Value,     // callable (Smi handle or TAG_FUNC)
    reject: Value,      // callable
}
```

### GC scanning

For `TAG_PROMISE`:
- Forward `result` (Value field)
- Forward each reaction in `PromiseReactionArray` (handler + capability fields)
- `scan_end` = `GcHeader + 12 + 8 + 8 + 8 + 8` = 44 bytes, aligned to 48

## 3. Microtask Queue

### Location

On `Vm`:

```rust
pub struct Vm {
    // ...
    microtask_queue: Vec<Microtask>,
    is_draining_microtasks: bool,  // guard against re-entry
}
```

```rust
struct Microtask {
    callback: Value,  // callable (onFulfilled or onRejected)
    argument: Value,  // the value or reason to pass
}
```

### Execution model

The microtask queue drains **after each synchronous task completes**:

1. `execute()` runs a script/function call synchronously
2. After `execute()` returns (either normally or with a pending Promise), drain the microtask queue
3. Each microtask: call `callback(argument)` — the callback is expected to return a Promise that may enqueue further microtasks
4. Drain continues until the queue is empty
5. If any microtask throws, reject the associated Promise and continue draining (per spec, microtask errors do not propagate to the macrotask)

### Where drain happens

Two entry points:

1. **`Context::eval()`** — after `vm.execute()` returns, call `vm.drain_microtask_queue()`
2. **Inside the VM's pending-callback pattern** — when a Promise's `resolve`/`reject` is called by a builtin, the reaction callbacks are enqueued as microtasks. These are drained after the current synchronous task completes (not inline).

### Microtask vs callback state machine

The existing `pending_array_op`, `pending_call`, `pending_assert` are **synchronous re-entrant state machines** — they pause the current execution, run a callback, and resume. They are not microtasks.

Promise reactions are **asynchronous** — they are scheduled to run after the current task completes. They use the microtask queue, not the pending-callback pattern.

Key distinction:
- Synchronous callback (array methods, Function.call): `pending_call` — immediate re-entrant
- Asynchronous callback (Promise.then): `microtask_queue` — deferred to after task completion

## 4. Promise Constructor (sync builtin)

`new Promise(executor)` and `Promise(executor)`:

```
1. Create new TAG_PROMISE object (state = pending, result = undefined)
2. Create resolve/reject functions (closure over the promise via env)
3. Call executor(resolve, reject) synchronously
4. If executor throws, reject the promise with the thrown value
5. Return the promise
```

### Resolve function (builtin)

```
1. If promise state ≠ pending, return
2. If resolution === promise, reject with TypeError (self-resolution)
3. If resolution is a TAG_OBJECT with .then property:
   - Call resolution.then(resolve, reject) — unwrap thenable
4. Else:
   - Set state = fulfilled, result = resolution
   - Trigger reactions (enqueue microtasks)
```

### Reject function (builtin)

```
1. If promise state ≠ pending, return
2. Set state = rejected, result = reason
3. Trigger reactions (enqueue microtasks)
```

## 5. `.then()` — the callback state machine extension

`.then(onFulfilled, onRejected)` is the key integration point with Rune's existing callback architecture.

### Simple case: promise already settled

```
1. Create new PromiseCapability (new Promise + resolve/reject)
2. Wrap onFulfilled/onRejected:
   - onFulfilled: call original, return value → resolve(new_promise, return_value)
   - onRejected: call original, return value → resolve(new_promise, return_value), or if throws → reject(new_promise, thrown)
3. Enqueue microtask with wrapped handler + promise result
4. Return capability.promise
```

### Complex case: promise pending

```
1. Create new PromiseCapability
2. Create PromiseReaction with wrapped handler + capability
3. Append to promise.reactions
4. Return capability.promise
```

### Wrapped handler

The wrapping is the key architectural challenge. The wrapping logic is:

```
function wrapCallback(handler, capability) {
    return function(value) {
        try {
            var result = handler(value);
            capability.resolve(result);
        } catch (e) {
            capability.reject(e);
        }
    };
}
```

This wrapping must be implemented as a **builtin** (not JS code). The wrap function:
1. Takes `handler` and `capability` (both Value)
2. When called (via the microtask drain loop), calls `handler(value)` synchronously
3. If handler returns normally, calls `capability.resolve(result)`
4. If handler throws, calls `capability.reject(thrown)`

This is implemented as a **dedicated builtin** that the microtask drain loop invokes, passing `(handler, capability, value)` as a 3-value Microtask payload.

## 6. Microtask Drain Loop (on Vm)

```rust
impl Vm {
    fn drain_microtask_queue(&mut self, gc: &mut SemiSpace) {
        self.is_draining_microtasks = true;
        while !self.microtask_queue.is_empty() {
            let tasks = std::mem::take(&mut self.microtask_queue);
            for task in tasks {
                // Call task.callback(task.argument)
                // If callback is a builtin wrapping handler, it will:
                //   1. Call the original handler
                //   2. Resolve/reject the capability's promise
                //   3. Potentially enqueue more microtasks
                let result = self.call_builtin(gc, task.callback, &[task.argument]);
                if let Some(exc) = self.pending_exception.take() {
                    // If the wrapping handler threw, reject the capability
                    // (This shouldn't happen if wrapping is correct, but be safe)
                    self.enqueue_reject_microtask(task.capability, exc);
                }
            }
        }
        self.is_draining_microtasks = false;
    }
}
```

## 7. `async`/`await` — Parser Desugaring

### `async function`

```
AsyncFunctionDeclaration:
  "async" "function" BindingIdentifier "(" FormalParameters ")" "{" FunctionBody "}"
  → FunctionDeclaration with is_async: true flag on Func
```

The parser sets `func.is_async = true`. At runtime:
- `MakeFunction` checks `is_async`
- When called, create a new Promise immediately
- Start executing the function body
- If the function returns (synchronously), resolve the promise with the return value
- If the function throws, reject the promise

### `await expr`

```
AwaitExpression:
  "await" UnaryExpression
  → Expr::Await(Box<Expr>)
```

The emitter transforms `await expr` into:

```
1. let _$promise_$ = expr;  // evaluate the expression
2. if _$promise_$ is not a Promise, wrap in Promise.resolve(_$promise_$)
3. Return a Promise and suspend the current function
4. When the promise settles, resume execution with the value
```

### Suspension mechanism

`await` requires suspending the function and resuming later. This is the **generator pattern**:

- `async function` body becomes a **generator** internally
- Each `await` is a yield point
- When `await promise` is reached:
  1. `promise.then(resume_generator, throw_into_generator)`
  2. Suspend the function (yield control back to the event loop)
  3. When the promise settles, the `.then` callback resumes the generator
  4. Generator `return` value becomes the await result (or throw if rejected)

This means `async` functions use the existing `Generator` infrastructure:
- `generator.rs` already has `Resume` / `Yield` opcodes
- Each `await` becomes a `Yield` that yields the inner promise
- The `.then` callback calls `generator.next(value)` or `generator.throw(err)`

### Simplification: No generator reuse

For v0.5, `async` generators (`async function*`) are out of scope. Plain `async function` is the target. The generator infrastructure is reused but the `async` function is always a one-shot generator:
- Created when called
- Yielded at each `await`
- Resumed by the promise's `.then`
- Completed when the function body finishes
- The wrapping Promise resolves/rejects with the generator's final return/throw

## 8. Interaction with existing mechanisms

### GC roots

When a Promise is created:
- Its reactions array is rooted (promise value is on the stack or in a local)
- Microtasks are rooted (they're in `Vm::microtask_queue`)
- During `drain_microtask_queue`, each microtask's values are on the stack

### Microtask queue and pending_array_op

- These are **independent**. `pending_array_op` is synchronous — it pauses and resumes within a single `execute()` call.
- Microtasks run **after** `execute()` returns.
- If a callback inside `pending_array_op` resolves a Promise, the reaction is enqueued but not drained until the current task completes.

### Microtask queue and exceptions

- If a microtask callback throws, the wrapping handler catches the error and rejects the capability's Promise (per spec, microtask errors don't propagate to macrotasks).
- The drain loop catches exceptions from each microtask, rejects the associated Promise, and continues.

### Week 1: Promise core + microtask queue
1. `TAG_PROMISE` + `PromiseReaction` types in `rune_core`
2. GC scanning for TAG_PROMISE
3. `Promise.resolve`/`Promise.reject` builtins (simple)

6. Microtask queue on Vm + drain loop

### Week 3: Testing + stabilization

## 10. Open questions

1. **Thenable unwrapping depth limit**: Spec requires recursive thenable unwrapping with infinite depth. Need a depth counter (e.g., 1000) to detect cycles. Implementation: resolve builtin checks `then` property on the resolution value and creates a new reaction.

2. **Unhandled rejection tracking**: Node logs a warning for unhandled rejections. v0.5 can skip this (defer to v0.6 debugger).

3. **`setTimeout(fn, 0)`**: Not a Promise feature but needed for some patterns. Can be a simple builtin that pushes to the microtask queue. Or defer to `queueMicrotask` builtin.

4. **`async` generator stack**: When `await` suspends an async function inside a deeply nested call (e.g., `a(await b(c(d(e))))`), the entire call stack must be captured in the generator state. This is handled by the existing Frame stack in Generator.

5. **Microtask queue vs pending_call**: If a `.then` callback is enqueued as a microtask while a `pending_array_op` is active, the microtask should NOT drain until the array operation completes. The drain loop must check that no synchronous state machines are active.

---

**Decision:** Review this design doc before implementing. The key risk areas are:
- Microtask queue integration with the existing synchronous execution model
- Thenable unwrapping depth
- `await` suspension mechanics (generator reuse)
- GC rooting of pending reactions and microtasks
