# v0.3 Copy-and-Patch: Strategic Design Decision

**Status**: A1c complete — LoadSmi ported behind `--stencil-jit` flag
**Author**: Agent, based on session discussion and literature review
**Date**: 2026-06-27 (updated 2026-06-27, A1c integration committed)

## 1. Problem

The current baseline JIT (`codegen_aarch64.rs`, 2124 lines) uses hand-rolled
instruction encoding to emit AArch64 machine code at runtime. This is the
dominant source of JIT bugs — every P-level bug in v0.1-v0.2 was an encoding
error (wrong shift, wrong mask, wrong opcode bit). The whitelist-drift bug
(P25) and the smi-range bug (P26) are recent examples.

The copy-and-patch paper (Hölzl et al., 2020) promises to fix this class of
bugs by replacing runtime instruction encoding with build-time compilation:
compile C functions ("stencils") with Clang/LLVM at build time, then emit them
at runtime via memcpy + hole patching. LLVM generates the machine code, not
the JIT author.

## 2. The mistake we almost made

The initial `build.rs` for `rune_jit_stencils` used a hand-rolled AArch64
encoder to generate stencil bytes. This duplicated the 2124-line encoder into
a second source of truth without delivering any of copy-and-patch's benefits:

| Aspect | Hand-rolled stencils (rejected) | Clang-compiled stencils (proposed) |
|--------|-------------------------------|-----------------------------------|
| Code quality | Identical to existing codegen | LLVM-optimized (scheduling, peephole) |
| Bug surface | Two encoders to maintain | One source of truth (C code) |
| Maintenance | Need AArch64 encoding skill | Write C, Clang generates bytes |
| Value over status quo | Zero (same bytes, more code) | Eliminates encoding bug class |

**Decision: Hand-rolled stencil generation is rejected.** The `build.rs`
encoder will be removed. Stencils will be compiled from C by Clang.

## 3. Architecture

### 3.1 Stencil C API

Each stencil is a C function in `crates/rune_jit_stencils/stencils/*.c`.
Stencils use **runtime helpers** for JIT-specific operations (stack push/pop,
register access, GC barriers). The helpers are compiled C functions that use
inline assembly to interact with JIT special registers (x22 = JIT stack,
x19 = VM, x20 = GC):

```c
// runtime.h — declares runtime helpers that stencils call.
// Helpers are compiled separately (rune_push.c). Their prologue/epilogue
// is stripped by build.rs — only the body (inline asm) is preserved.

#include <stdint.h>

// Push val onto the JIT stack at x22, then advance x22.
// Body (after stripping prologue/epilogue): STR x0,[x22]; ADD x22,x22,#8 (8 bytes)
void rune_push(int64_t val);
```

### 3.2 Stencil C functions

Each stencil is a C function that calls runtime helpers:

```c
// load_smi_16.c
#include "runtime.h"

// LoadSmi(imm): push Smi(imm) onto JIT stack.
// Compiled as: MOVZ W0, #0xDEAD ; B _rune_push (tail-call optimized)
// The B is a link hole resolved at JIT time.
void load_smi_16(void) {
    rune_push(0xDEAD);   // 0xDEAD is a placeholder — patched at JIT time
}
```

The compiler generates:
```
movz w0, #0xDEAD       (or movz + movk for >16-bit)
b    _rune_push         (tail-call to helper — no ret needed)
```

The `rune_push` helper is compiled separately and emits:
```
stp  x22, x21, [sp, #-16]!   // prologue (stripped)
str  x0, [x22]                 // body (preserved)
add  x22, x22, #8              // body (preserved)
ldp  x22, x21, [sp], #16      // epilogue (stripped)
ret                            // epilogue (stripped)
```

### 3.3 Build-time pipeline

`build.rs` does:

1. **Compile** each `.c` file with `clang -O2 -c -ffreestanding -target arm64-apple-macos`
   via `std::process::Command`.

2. **Parse** the `.o` file (Mach-O) manually to extract:
   - `__TEXT,__text` section bytes (function body)
   - Relocation entries: `ARM64_RELOC_BRANCH26` for B/BL link holes
   - Instruction-template verification via bitmask matching (build fails if Clang output changes)

3. **Generate** Rust code embedding:
   - Helper byte arrays (body only — prologue/epilogue stripped)
   - Stencil byte arrays (value holes + link holes included)
   - Hole descriptors:
     ```rust
     HoleDef { byte_offset: 0, bit_offset: 5, bit_width: 16 }  // MOVZ imm16
     LinkHoleDef { byte_offset: 4, helper_name: "rune_push" }  // B _rune_push
     ```

#### 3.3.1 Identifying holes

The challenge: given a compiled function body, how do we know which bytes are
patchable holes?

**Option A: Relocation analysis** (preferred). Clang emits relocations for:

- Branch targets: BL/B instruction's 26-bit offset → `R_AARCH64_CALL26`
- Data symbols: ADRP+LDR pair → `R_AARCH64_ADR_PREL_PG_HI21` + `R_AARCH64_LDST64_ABS_LO12_NC`
- If we encode the stencil operand as an immediate that appears as a literal
  pool entry, the relocation tells us where it is

For immediate operands (like `smi_val` in `movz x0, #smi_val`): the compiler
encodes the immediate directly in the instruction word. There is no relocation
for an immediate — it's just bits in the instruction. We need another way to
find it.

**Option B: Parameter annotation** (prototype). Use the C preprocessor to
stamp known marker values into the stencil, then scan for them:

```c
#define STENCIL_MARKER 0xDEADBEEF

void stencil_load_smi(int64_t smi_val) {
    // Force smi_val to appear as a MOVZ immediate
    // The marker value will be patched at runtime
    rune_push(STENCIL_MARKER);
    (void)smi_val;  // smi_val replaces STENCIL_MARKER in patch
}
```

Then `build.rs` scans the function bytes for the marker `0xDEADBEEF` (or its
encoded form) to find the hole position.

**This is fragile and rejected.** The marker might be encoded differently
by different Clang versions, or optimized away.

**Option C: Section-based parameter passing** (proposed). Put each patchable
operand in a designated section:

```c
__attribute__((section("__TEXT,stencil_hole")))
const int64_t __hole_smi_val = 0;

void stencil_load_smi(void) {
    // Load the hole value and push it
    register int64_t val asm("x0");
    __asm__("ldr x0, [%[hole]]" : : [hole] "r"(&__hole_smi_val));
    rune_push(val);
}
```

The relocation for `__hole_smi_val` tells us where the LDR instruction is.
At runtime, we write the actual `smi_val` into the hole data slot.

**This modifies the stencil body** (adds a load from memory) which LLVM may
or may not optimize depending on the architecture. On arm64, LDR from a
literal pool is typical and the compiler handles it.

**Option D: Position-based hole detection** (recommended for prototype).
Known structure: stencils follow a fixed pattern. For `stencil_load_smi`:

- Instruction 0: MOVZ x0, #imm16 (or MOVZ + MOVK for 32-bit)
- Instruction 1: STR x0, [x22]
- Instruction 2: ADD x22, x22, #8
- Instruction N: RET

`build.rs` disassembles the function body (or uses the known instruction
positions from the compiled output) and identifies holes by instruction
position. The RET at the end marks the boundary for prologue/epilogue
stripping.

**This is architecture-specific** but predictable because:
- `-ffreestanding` prevents compiler from inserting runtime init calls
- `-O2` gives deterministic instruction selection for simple patterns
- We verify at build time that the bytes match expected encodings

If Clang's output doesn't match expectations, the build fails — preventing
silent regressions.

**Decision: Option D for value holes, Option A for link holes.**
Value holes (MOVZ imm16) use instruction template matching (Option D) with
build-time verification. Link holes use relocation analysis (Option A) —
Mach-O ARM64_RELOC_BRANCH26 is parsed from the `.o` file.

### 3.4 Runtime pipeline

At JIT compile time:

1. **Emit runtime helpers once** — shared by all stencil instances:
   - `rune_push` body
   - `rune_pop` body
   - Other helpers as needed

2. **For each opcode in the bytecode sequence:**
   a. Look up the stencil by opcode
   b. Patch value holes with the bytecode operand (e.g., `smi_val`)
   c. Patch link holes with the correct relative offset to the runtime helper
   d. Emit the patched stencil into the JIT buffer

3. **Prologue + epilogue** are emitted as separate stencils (with no patches)

### 3.5 Calling convention

Stencils execute in the JIT's register convention:
- x19 = VM pointer (read-only during stencil execution)
- x20 = GC pointer (read-only)
- x21 = LOC_REG (read-only)
- x22 = JIT stack base (read-write — modified by push/pop)
- x23 = saved LOC_REG (for inlining)
- x0-x7 = scratch/temporaries
- x8 = indirect result register
- x9-x17 = scratch (caller-saved in ARM64 ABI)

Runtime helpers preserve callee-saved registers (x19-x29) as required by
the ARM64 ABI.

## 4. Prototype plan

### 4.1 Key risk: Clang output unpredictability

The single biggest risk: Clang may produce different instruction sequences
for the same C code across versions, optimization levels, or target triples.
If the generated bytes are unpredictable, hole detection fails.

**Risk mitigation:** The prototype compiles one stencil, disassembles its
bytes, and verifies against expected instruction templates. If the template
doesn't match, the build fails with a clear error showing the actual bytes.

### 4.2 Prototype steps

1. **Write `prototype.c`** — a single C stencil with a runtime helper call
2. **Compile with `cc` crate** → get `.o` file bytes
3. **Parse with `object` crate** → extract function symbols, relocations
4. **Extract stencil body bytes** — strip prologue (STP) and epilogue (RET)
5. **Identify hole positions** — use instruction template matching
6. **Write validation test** — patch the stencil, verify against known encoding
7. **Measure** — `build.rs` compile time, stencil size, patch throughput

### 4.3 Success criteria

Prototype is successful if:
- Clang produces deterministic output for the same source (verified by
  checksum)
- The extracted bytes match expected instruction templates
- Hole patching produces correct machine code (verified by test)
- Build time overhead is <1s (stencil compilation is incremental)

### 4.4 Failure modes

If the prototype fails:
- Clang output is non-deterministic between builds
- Instruction template matching is fragile across Clang versions
- The `.o` parsing is too complex for the value delivered

**Fallback:** Abandon copy-and-patch rewrite for v0.3. Spend remaining weeks
on Float Self-Tagging (Phase B) and Nofl GC (Phase C). Revisit at v0.4
with a revised approach.

## 5. Current state

**A1c complete** — `rune_jit_stencils` crate builds real C stencils at compile time,
and `codegen_aarch64.rs` uses them behind the `--stencil-jit` flag:

- `build.rs` compiles C stencils + helpers with Clang (`-O2 -ffreestanding`), extracts bytes
- Value holes (MOVZ/MOVK imm16) identified by instruction template bitmask matching
- Link holes (ARM64_RELOC_BRANCH26) found by parsing Mach-O relocations in the `.o` file
- 3 real C stencils (`load_smi_16`, `load_smi_32`, `load_const`) + 3 naked-asm stencils (`push_reg`, `pop_reg`, `ret`)
- JIT integration (`codegen_aarch64.rs:780-809`): LoadSmi ported behind `--stencil-jit` flag
  - **Inline approach** (not shared helper): emit MOVZ/MOVK stencil bytes, patch value holes
    with 64-bit MOVZ/MOVK (sf=1), append STR+ADD inline from helper body
  - Old codegen kept as fallback until all 57 opcodes ported
- CLI flag `--stencil-jit`/`--no-stencil-jit` (default false) wired through Context → Vm → CodeGen
- Behavioral test: 4 cases (42, add-chain, -1, max Smi) tested with both flags, assert equal results
- 67 stencil + 48 baseline + 317 integration tests pass, clippy-clean
- Hand-rolled AArch64 encoder removed from build.rs

**Key design decision — inline vs shared:** The copy-and-patch paper uses shared helpers
(call via B/BL, one helper for many stencils). For LoadSmi, the "helper" is 8 bytes (STR+ADD).
A shared helper would add BL+RET = 8 bytes overhead to save 8 bytes — a wash with worse icache.
**Inlining is correct for this opcode.** Per-opcode decisions documented in each stencil's C file.

## 7. Stencil rule (enforced)

**Stencils with patchable values or non-trivial logic must be real C.**
Stencils that are pure fixed-instruction sequences with no patchable fields
may use naked asm.

| Category | Approach | Examples |
|----------|----------|----------|
| Value holes + link holes | Real C (`rune_push(0xDEAD)`) | `load_smi_16`, `load_smi_32`, `load_const` |
| Fixed instructions, no immediates | Naked asm | `push_reg`, `pop_reg`, `ret` |

Enforcement: `path_a.rs` contains a test per real-C stencil that compiles the
actual C file and verifies encoding. A naked-asm regression on a real-C
stencil would fail that test. Future stencils must add a parallel test.

  Inventory: 3 real-C stencils, 3 naked-asm stencils (trivial only).
